# PT Orchestrator — pt-orc
*L1 — always loaded, never dropped during an engagement session*

---

## MRK:PTORC_ROLE — Role | ptorc,role | L38-48

This skill is the single entry point for all PT work. It:
1. Reconstructs engagement context from the state snapshot
2. Determines current position and next action
3. Dispatches the correct L2 skill for the requested work
4. Receives L2 output and updates the state snapshot
5. Manages return passes to earlier stages without losing progress

---

## MRK:PTORC_PRINCIPLES — Core Principles (Immutable) | ptorc,principles,core,immutable | L49-61

1. **Evidence-first.** Every finding traces to at least one captured artefact on disk. No finding is written without evidence.
2. **Three evidence streams:** MSF DB (queryable source of truth) · Automated raw logs (tool output under `evidence/`) · Manual analyst input (terminal sessions, curl, dig). All three maintained and correlated.
3. **MSF DB with caveats.** When DB conflicts with observed evidence, raw log wins. Flag with `notes -a -h <IP> -t db.caveat`.
4. **Validation pipeline:** `detected → suspected → validated → high_confidence`. Confidence maps: UNCERTAIN→suspected, LIKELY→validated, CERTAIN→high_confidence.
5. **Confirmed vs. precautionary.** Confirmed = exploited or impact demonstrated. Precautionary = banner/version matches CVE, not exploited. Mark `[Precautionary]`, confidence LIKELY.
6. **Finding vs. Observation.** Every F-XX requires a Recommendation. OBS-XX is informational only — no Recommendation ever.
7. **Versioning.** Every change increments version in filename AND internal field. Never reissue same version number.
8. **Evidence-based only.** No speculative content. No invented findings. CVE references where applicable.

**Directory rules:**
- `evidence/_sweep/` — raw scan files only (`.nmap`, `.xml`, `.gnmap`, per-session logs)
- `evidence/_dns/` — written by `01_dns_recon.sh`; contains crt.sh, subfinder, amass, httpx, takeover candidates
- `evidence/_ip_analysis/` — written by `02_ip_range_analysis.sh`; per-IP subdirs `<IP_safe>/` (dots→underscores) with WHOIS, BGP, geo, traceroute, Shodan; consolidated `ip_range_report_<TS>.md` at root
- `evidence/_wpscan/` — written by `06_wpscan.sh`; per-target subdirs `<label>/` (URL sanitized) with WPScan output; consolidated `wpscan_report_<TS>.md` at root
- `working/` — all analyst-facing summaries, follow-up lists, target lists, state snapshots
- `evidence/<IP>/` — all per-host tool output and manual session logs
- Manual session logs are generated by Claude from pasted terminal output — never created manually

**MSF DB connection paths (two independent paths):**
- `msfconsole` / `export_db` — Ruby pg adapter, Unix socket; always works on Kali
- `db_query` / `psql` — direct psql; fails if PostgreSQL only listens on Unix socket. `parse_db_conf` tests TCP then Unix socket; sets `DB_DIRECT_AVAILABLE=0` if both fail. Enum phases fall back to `services_tcp_*.csv` already on disk.

## MRK:PTORC_DISPATCH — Dispatch Table — L2 Skills | ptorc,dispatch,table,l2,skills | L157-178

L1 reads the request, maps it to an L2, passes the required context, and expects a defined output.

| L2 Skill | Triggers | Pass in from snapshot | Expect back |
|---|---|---|---|
| `pt-recon` | S1/S2 work; "sweep", "OSINT", "DNS", "subdomain", "IP analysis", "surface", new target range | Scope (subnets/domains), tester IPs, MSF workspace, existing host list | Updated host/service inventory, evidence paths, findings stubs |
| `pt-enum` | S3/S4 work; named service ("SMB", "NFS", "SNMP", "TLS", "HTTP", "FTP", "WordPress", specific IP/port), "enumerate", "deeper look" | Target IP, **mode (PTI/PTE)**, **stealth tier**, known open ports/services, existing findings for that host | Evidence paths, findings stubs, validation state updates |
| `pt-evidence` | "process evidence", pasted tool output, "log review", "what does this mean", S6 | Raw output (pasted or file ref), existing findings index | Structured finding stub (ready to append to Findings .md) |
| `pt-report` | "write finding", "draft report", "executive summary", "section", S7 | Findings index, scope, engagement metadata, severity distribution | Finding body text, section text, report-ready content |

### Dispatch Protocol

When a request maps to an L2:
1. State what you are doing: *"Loading pt-enum for SMB enumeration on <target-IP>."*
2. Extract the required fields from the active state snapshot.
3. Invoke the L2 mentally (treat its SKILL.md content as active instructions).
4. On completion, receive L2 output and update the state snapshot.
5. State that L2 work is complete and context is released: *"pt-enum complete for this host. State snapshot updated."*

## MRK:PTORC_RETURNPASS — Return Pass Protocol | ptorc,returnpass,return,pass,protocol | L179-196

A return pass occurs when work on an earlier stage is needed after that stage was marked COMPLETE.

**Triggers:** New finding implies re-checking earlier recon data · Client provides additional info · Evidence gap discovered during report writing · Specific host/service needs deeper look after initial pass.

**On return pass initiation:**
1. Identify the target stage (e.g. S3).
2. State: *"Opening Return Pass 1 on Stage 3 — [reason]."*
3. Append to state snapshot Stage Registry: `S3: COMPLETE+ | RP1 | <date> | <reason>`
4. Dispatch appropriate L2 with: (a) existing evidence/findings for relevant hosts, (b) specific objective of this pass, (c) "return pass" flag so L2 doesn't repeat completed work.
5. On completion, append RP summary to state snapshot under that stage's entry.
6. Resume from prior current position.

**Rule:** Return pass does not reset stage status. `COMPLETE+` means complete with additional passes — the `+` count is implicit from RP entries.

## MRK:PTORC_PITFALLS — Common Pitfalls | ptorc,pitfalls,common | L444-471

| Pitfall | Correct approach |
|---|---|
| Evidence not linked to target IP | Every file in `evidence/` under `<IP>/` or named `_` dir |
| MSF DB accepted as ground truth | Flag misidentifications; raw log wins on conflict |
| High from single stream | Require two streams or active confirmation |
| Precautionary CVE without label | Add `[Precautionary]`; set confidence LIKELY |
| Finding downgraded without rationale | Add explicit note to finding body |
| Subnet bridge from one indicator | Require sequential MACs + identical interface stats |
| Session ended without snapshot | Generate snapshot before closing — always |
| Same version number reissued | Increment before saving; filename must match internal field |
| Finding without Recommendation | Every F-XX requires Recommendations section |
| Recommendation on Observation | OBS-XX informational only; promote to F-XX if action needed |
| Workspace not verified at session start | `msf6 > workspace` before any command |
| Return pass wiping prior progress | COMPLETE+ preserves history; RP appends, never resets |
| 02 run after 03 starts scanning | 02 is passive OSINT only — must run before 03 sends any packet |
| 06 run without wp_targets.txt | 05 populates it; or use `06 --detect` to run detection sweep first |
| `--yes` used for PTE scan | Never skip PTE scope confirmation; visual confirm required every run |
| WordPress finding without 06 evidence | Run 06 after 05 to get structured WPScan output before writing findings |
| "No hosts in DB" during enum/udp | Check DB_DIRECT_AVAILABLE — psql may be failing (Unix socket vs TCP). CSV fallback activates automatically; enum proceeds from services_tcp_*.csv |
